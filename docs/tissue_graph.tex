\begin{abstract}
Living tissues couple biochemical signalling, cellular interaction, mechanical
force, and collective dynamics across scales. The frameworks built so far
(\textsc{CellGraph}, \textsc{NeuralGraph}, \textsc{MPMGraph},
\textsc{MetabolismGraph}) each implement \emph{one} graph type; a tissue needs
them coexisting. This document is a single blueprint, distilled from a working
prototype, for building one repository that unifies them. It has three parts:
(i) the missing primitive --- a \textbf{hierarchical graph container} stated in
the language of \textbf{sets} and \textbf{operators}; (ii) the \textbf{contract}
that lets a large language model (LLM) drive a registry of low-level operators
through a declarative \emph{simulation} specification; and (iii) the
\textbf{experiment} --- ten distinct simulations and an emergent ant-trail model,
all produced from the same code by changing only the simulation file --- with the
lessons that follow for how to construct the repository.
\end{abstract}

\section{Two entities: sets and fields}

The container is built from two kinds of object: \textbf{sets} and \textbf{fields}.
A \textbf{set} $S_k$ is a discrete collection of like entities at scale $k$ ---
metabolites, particles, cells, populations, organisms (Fig.~\ref{fig:hier}).
Adjacent scales are linked by \textbf{parent maps}
\[
  \pi_k : S_k \longrightarrow S_{k+1},
\]
where $\pi_k(x)$ is the higher-level object that contains $x$. A cell is built from
two kinds of $S_0$ entity --- its \textbf{particles} (mechanics) and its
\textbf{metabolites} (chemistry) --- and $\pi_0$ sends each of them to the cell it
belongs to, so $\pi_0(x)=c$ reads ``$x$ belongs to cell $c$''. The constituents of a
given cell $c$ are its \emph{fibre}, the set of everything that maps to $c$,
\[
  \pi_0^{-1}(c)=\{\,x\in S_0 : \pi_0(x)=c\,\},
\]
so a cell simply \emph{is} the collection of its particles and metabolites.
A \textbf{field} $f:\Omega\to\mathbb{R}^q$ is a continuous quantity defined on its
own grid --- morphogen concentration, pressure, substrate stiffness, and so on.
Fields are \emph{not} part of the containment hierarchy: they neither contain nor
are contained by cells. Instead, each field is bound to exactly one set level
through an \emph{exchange} relation (particle--grid transfer, cell sampling, or
population-level secretion and sensing); these couplings are the operators of
Section~\ref{sec:operators}.

\section{Four operators}
\label{sec:operators}

A GNN is an \textbf{operator} $\mathcal{O}$ returning a time-derivative
contribution, dispatched by the relation it acts on. We use four kinds, one per
relation the container admits (Fig.~\ref{fig:ops}):
\begin{align*}
\textbf{Lateral}\quad   & \mathcal{O}_E,\ E\subseteq S_k\times S_k
  &&\text{(ODE interaction $+$ graph Laplacian)}\\
\textbf{Aggregate}\ \uparrow\quad & \textstyle\sum_{\pi}: S_k \to S_{k+1}
  &&\text{(reduction over fibres of $\pi$)}\\
\textbf{Broadcast}\ \downarrow\quad & \pi^{*}: S_{k+1}\to S_k
  &&\text{(lift along $\pi$)}\\
\textbf{Exchange}\quad  & \mathcal{S},\mathcal{G}: S_k \leftrightarrow F
  &&\text{(scatter / gather; bipartite)}
\end{align*}

\noindent Two pieces of notation: $S_k\times S_k$ is the set of all node \emph{pairs}, so an
edge set $E\subseteq S_k\times S_k$ is simply a graph on $S_k$ --- which nodes
interact (e.g.\ neighbours within a cutoff radius, or a fixed connectome). And
$\mathcal{S},\mathcal{G}$ are the two halves of a field coupling: \emph{scatter}
$\mathcal{S}:S_k\to F$ writes object state into the field, while \emph{gather}
$\mathcal{G}:F\to S_k$ reads it back. Each operator in turn:

\paragraph{Lateral $\mathcal{O}_E$.} Message passing \emph{within} one scale along
the edges $E$. It carries the within-scale physics: pairwise interactions (forces,
adhesion, signalling) plus a graph Laplacian that diffuses or smooths quantities
over the edges. Examples: particle--particle elasticity, cell--cell flocking or
adhesion, neuron--neuron signal propagation.

\paragraph{Aggregate $\textstyle\sum_\pi$ (up).} Pools each parent's fibre into one
quantity --- a cell's position is the mean of its particles, its metabolic state the
sum over its molecules. This is how fine-scale dynamics are summarised into, and so
move, the coarser object above.

\paragraph{Broadcast $\pi^{*}$ (down).} The adjoint of Aggregate: it copies a
parent's state to every child in its fibre, so a decision taken at the cell level
(a body force, a polarity, a signalling output) is applied identically to all of
that cell's particles. Aggregate and Broadcast are a matched up/down pair on the
same map $\pi$.

\paragraph{Exchange $\mathcal{S},\mathcal{G}$.} Couples a set to a field: scatter
$\mathcal{S}$ deposits/secretes object state into the field, gather $\mathcal{G}$
samples/senses the field back onto the objects. The relation is bipartite
(objects~$\leftrightarrow$~grid); it is exactly the MPM particle--grid transfer
(P2G/G2P) and chemotactic secretion/sensing.

\section{Dynamics: a schedule}
\label{sec:schedule}

A simulation is a multi-rate composition of operators, declared as a
\code{Schedule}. In math terms, one tick is the composition
$\Phi=\mathcal{O}_n\circ\cdots\circ\mathcal{O}_1$, and the dynamics is its
iteration $x_{t+1}=\Phi(x_t)$ --- a discrete flow (with ``multi-rate'' meaning
some operators fire only on every $k$-th tick). The \code{Schedule} is just the
code-level encoding of that composition. Each operator is \emph{pure}: instead of
editing the state in place, it only \emph{returns its contribution} --- a delta
$\Delta_i$ (a time-derivative term) for the sets it touches. The schedule sums the
deltas acting on each set and integrates once per tick,
$x_{t+1}=x_t+\Delta t\sum_i\Delta_i$. Two consequences follow: several operators may
act on the same set in one tick (their deltas simply add --- none overwrites
another), and because nothing is mutated in place, gradients flow through the sums
and the integrator, so the whole rollout stays differentiable. The LLM searches over
schedules and operator parameterizations --- one well-defined action space instead
of per-domain code.

\section{Two layers and the contract between them}

The prototype's purpose was not the tissue physics but figuring out \emph{how an
LLM should drive a registry of low-level operators}. The answer is a strict
two-layer split that meets only at one declarative, validated file --- the
\textbf{simulation spec}. In effect this is a small domain-specific language: the
spec is the \emph{source}, the registered operators are the \emph{primitives}, and
the engine is the \emph{interpreter}. The \code{simulation.yaml} is therefore not a
simulation but a \emph{specification} of one; the simulation is the run the engine
produces from it.

\begin{center}
\renewcommand{\arraystretch}{1.25}
\begin{tabular}{@{}p{3.0cm}p{11.5cm}@{}}
\toprule
\textbf{INTENT} & natural language (``2000 cells, two mechanical types, a chemical that guides only one'')\\
\quad$\downarrow$ \emph{LLM compiles} & bounded, schema-checked generation\\
\textbf{SPEC} & \code{simulation.yaml} --- sets $\cdot$ fields $\cdot$ operators($+$selectors) $\cdot$ schedule\\
\quad$\downarrow$ \emph{engine resolves} & names looked up in the registry\\
\textbf{REGISTRY} & \code{@register\_operator} / \code{@register\_field} --- the only place code lives\\
\quad$\downarrow$ & \\
\textbf{ENGINE} & generic: build \code{Hierarchy} $\to$ run \code{Schedule} $\to$ \code{zarr} (written once)\\
\bottomrule
\end{tabular}
\end{center}

The \textbf{spec is the contract}. The LLM owns everything above it (intent $\to$
spec); the code owns everything below (operators). They never touch directly. This
makes the system both LLM-drivable and human-auditable: the spec is small, typed,
and reviewable, unlike the 500-line flat \code{config.py} it replaces. Three properties make the contract enforceable, and
they answer the three questions this experiment was built around
(Table~\ref{tab:modes}).

\section{The simulation specification}

A simulation is built from one vocabulary: \textbf{sets} (a leaf set declares its
\code{parent} and \code{per\_parent}), \textbf{fields} (each \code{couples\_to} one
set), \textbf{operators} (each acting at a \emph{selector} \code{set} or
\code{set[attr=value]}), and a \textbf{schedule} (the per-frame ordered list).
Listing~\ref{lst:agg} is a complete example.

\begin{lstlisting}[style=yaml,caption={A complete simulation: soft cells secrete and follow a chemical and self-aggregate.},label=lst:agg]
name: aggregate
seed: 0
n_frames: 250
sets:
  cell:
    n: 100
    types: {soft: {fraction: 0.5, youngs: 12}, stiff: {fraction: 0.5, youngs: 300}}
  particle: {parent: cell, per_parent: 100, radius: 0.02}   # containment
fields:
  chemical: {frame: grid, res: 96, diffusion: 0.1, decay: 0.05, couples_to: cell}
operators:
  - {op: motility, at: cell, speed: 15.0, rot: 0.4}
  - {op: mpm, at: particle, n_grid: 128, substeps: 8, a_max: 800, drag: 4}
  - {op: secrete, at: "cell[type=soft]", to: chemical, rate: 1.0}    # only pop 1 makes it
  - {op: sense,  at: "cell[type=soft]", from: chemical, gain: 150.0} # only pop 1 follows it
schedule: [aggregate, [motility], secrete, chemical.diffuse, sense, mpm]
\end{lstlisting}

The validator (prototype: \code{scenario\_schema.py}) guarantees that a file which
loads is runnable: every \code{op} exists; every \code{at}/\code{to}/\code{from}
references a declared set/field; type \code{fraction}s sum to one; every schedule
token resolves. (One trap: never use a YAML~1.1 boolean key such as
\code{on}/\code{off} --- we use \code{at}.)

\section{Natural language $\to$ simulation: the repeatable mapping}

To make the LLM's compilation \emph{repeatable} (same request $\to$ same
structure), it follows fixed rules; the most useful:

\begin{enumerate}\itemsep2pt
\item ``$N$ cells/agents'' $\to$ a set with \code{n: N}.
\item ``made of $K$ X per cell'' $\to$ a contained set with \code{parent}/\code{per\_parent} (two levels).
\item ``two types differing by $P$'' $\to$ \code{types:} with $P$ as a per-type property.
\item ``mechanical/elastic/rigid'' $\to$ particles $+$ \code{mpm}; map stiffness to \code{youngs}.
\item ``moves by boids / flock'' $\to$ \code{boids}; ``wanders / motile'' $\to$ \code{motility}.
\item ``creates/secretes a chemical'' $\to$ a \code{field} $+$ \code{secrete};
      ``follows / chemotaxis'' $\to$ \code{sense}; ``ant trails'' $\to$ \code{trail}.
\item \textbf{``only population X does Y''} $\to$ the selector \code{at: cell[type=X]}.
      This is the highest-leverage primitive: it scopes behaviour to a
      subpopulation and lets two behaviours coexist.
\end{enumerate}

\section{When the present code is not enough (a missing model)}

The pipeline fails loudly and points at the gap, e.g.\ for a request needing
field advection or reaction--diffusion:
\begin{lstlisting}[style=yaml]
operator 'reaction_diffusion' not in registry.
Available: ['adhesion','boids','mechanics','motility','mpm','secrete','sense','trail']
\end{lstlisting}
The fix is \textbf{one new \code{Operator} subclass} --- never an engine change:
write \code{forward(self, H, mask) -> {set: accel}} (or mutate a field, return
\code{\{\}}); decorate \code{@register\_operator}; declare its needs; reference it
in the simulation. We exercised this live twice during the experiment: adding
\code{adhesion} unlocked cell \emph{sorting}, and adding \code{trail} unlocked
ant \emph{trail-following} --- each $\sim$20 lines, with the engine and all other
simulations untouched.

\section{When a cell/particle property does not exist}
\label{sec:prop}

Operators \emph{declare} what they need (\code{REQUIRES\_PARAMS},
\code{REQUIRES\_TYPE\_PROPS}); the validator checks before running, resolving a
property along the containment chain (\code{mpm} acts on particles but
\code{youngs} lives on the parent cell's types):
\begin{lstlisting}[style=yaml]
# missing required property -> hard error, before any computation:
operator 'mpm' requires property 'youngs' on every type of 'cell'; missing on type 'soft'.
# property no operator reads -> warning (typo guard), run proceeds:
[warn] property 'viscosity' on cell.soft is read by no operator (known: ['fraction','youngs'])
\end{lstlisting}
\textbf{Design rule: capabilities are declared, not discovered at a crash.} The
operator that consumes a property is the single place that declares the
requirement, turning a would-be \code{AttributeError} thousands of substeps deep
into a one-line message up front.

\section{What the process taught us}

\begin{enumerate}\itemsep2pt
\item \textbf{The intermediate representation is the whole game.} Ten distinct
behaviours and an ant model were all different YAML over the same code.
\item \textbf{Selectors are the highest-leverage primitive.} \code{cell[type=soft]}
expresses ``only population~1 does~X'' and lets behaviours coexist; every operator
must honour the selector mask.
\item \textbf{Verification must check \emph{intent}, not just ``it ran''.} The hard
bugs were physical: unstable diffusion ($dt\!\cdot\!D>0.25$), wrong particle volume,
unclamped accelerations blowing up MPM, cell inversion at very low stiffness, GPU
non-determinism breaking reproducibility, and --- subtlest --- the \emph{wrong
metric} (net displacement vs.\ nearest-neighbour clustering).
\item \textbf{Honesty about regime.} Self-secreted chemotaxis aggregates only below
a density threshold (above it the field is spatially flat --- correct physics); the
simulation/verify output should state the regime, not silently show the one setting
that worked.
\item \textbf{Determinism is a feature you must force} (deterministic CUDA kernels),
or ``reproduce with a new seed'' is meaningless.
\end{enumerate}

\part*{Part IV --- Building the repository}
\addcontentsline{toc}{section}{Part IV --- Building the repository}

\section{Proposed methodology}

\paragraph{Layering (strict; no leaks across).}
\begin{itemize}\itemsep1pt
\item \code{models/base.py} --- entities, operator base, the capability-contract fields.
\item \code{models/registry.py} --- the only name$\to$code map (entity / operator / field).
\item \code{models/<domain>/*.py} --- operators, one concern per file, each declaring \code{REQUIRES\_*}.
\item \code{simulation.py} --- the schema $+$ validator (the contract gatekeeper).
\item \code{engine.py} --- generic; contains \textbf{no} simulation-specific logic.
\item \code{viz.py} --- generic over the \code{Hierarchy}/\code{zarr}; swappable backends (2-D now, 3-D later).
\item \code{archive.py} $+$ \code{simulations/} --- the growing, reproducible gallery.
\end{itemize}

\paragraph{The build loop (how LLM and code co-evolve).}
\begin{enumerate}\itemsep1pt
\item user states intent in natural language;
\item LLM compiles $\to$ \code{simulation.yaml} (following the mapping rules);
\item validator runs: unknown op $\to$ write one operator; missing property $\to$ fix
spec or add the requirement; else it is runnable;
\item engine runs; \textbf{verify} checks intent $+$ repeatability $+$ well-formedness;
\item \code{archive.py} records spec$+$seed$+$metrics$+$thumb into the gallery;
\item new capability $\to$ one new registered operator ($+$ its \code{REQUIRES\_*}),
never an engine edit.
\end{enumerate}

\paragraph{Invariants to hold the line.}
\begin{itemize}\itemsep1pt
\item Code lives only behind the registry; the engine never branches on simulation names.
\item Every operator honours its selector mask and declares its \code{REQUIRES\_*}.
\item Every simulation ships an intent check; ``it ran'' $\neq$ ``it worked''.
\item Archive the recipe (spec$+$seed); regenerate the dish.
\item When a result holds only in a regime, the spec/verify says so.
\end{itemize}

This is the contract that lets the LLM operate at the top (intent $\to$ simulation,
plus occasional single-operator authoring) while the low-level code stays a clean,
growing, verifiable registry underneath.
